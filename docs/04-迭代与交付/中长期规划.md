# 中长期开发规划

## 1. 规划口径

本文用于承接 Sprint 13 之后的中长期开发节奏。规划对象仍遵循本目录统一口径：

- Epic 表示能力域。
- Backlog 表示可跟踪交付项。
- Sprint 表示时间盒。
- Release 表示版本发布判断。

本规划不是冻结需求。每个 Sprint 开始前仍需按当前风险、验证结果和用户反馈做一次轻量取项确认。

## 2. 阶段目标

| 阶段 | 建议版本 | 建议 Sprint | 目标 |
| --- | --- | --- | --- |
| 发布收口 | V1.0 | Sprint 13 | 收口 Provider 诊断、V1.0 验收报告和质量回归最小模型 |
| 质量闭环 | V1.1 | Sprint 14 | 让评估结果能沉淀、导出、对比，并反向驱动配置优化 |
| 生产化接入 | V1.2 | Sprint 15 | 让真实 Provider 接入、凭据、复测和检索副本恢复可运维 |
| 知识库治理 | V1.2 | Sprint 16 | 提升文档质量、批量治理和知识库健康可见性 |
| 稳定性观测 | V1.3 | Sprint 17 | 让运行指标、慢链路、任务补偿和备份恢复能被持续验证 |
| 协作治理 | V1.3 | Sprint 18 | 支持配置发布、QA 协作、审计报表和有效权限解释 |

## 3. 未来 5 个 Sprint 主线

### Sprint 14：评估结果驱动配置优化

承接 Sprint 13 的 EvaluationRun / Result 最小闭环，重点补齐异步状态、失败重试、评估指标汇总、结果导出、Config Revision 对比和失败样本到配置草稿的优化入口。

成功标准：研发和测试能从一批评估结果定位失败样本、查看相关配置差异，并生成一个可复核的配置优化草稿。

### Sprint 15：真实 Provider 生产化接入

重点处理真实 LLM、Embedding、Rerank、Dense、Sparse、Graph Provider 的凭据配置、脱敏展示、连通性诊断、限流诊断和副本重建恢复。

成功标准：测试环境可以在不泄露凭据的前提下完成 Provider 复测，并把失败原因落到可排障的报告里。

### Sprint 16：知识库治理与文档质量

围绕文档解析质量、异常 Chunk、批量重解析、批量重建索引、知识库健康概览和 Chunk 排除标记展开。

成功标准：知识库负责人可以看到影响 QA 效果的文档质量问题，并能批量触发治理动作。

### Sprint 17：稳定性与观测

补齐 QARun、IngestJob、Provider 调用指标，增加慢链路诊断、错误摘要、健康面板、任务幂等和备份恢复演练。

成功标准：出现慢查询、Provider 失败或任务重复执行时，研发能通过平台数据定位问题，而不是只看后端日志。

### Sprint 18：协作与治理增强

面向多人协作补齐配置发布记录、变更说明、回滚确认、QA Run 分享评论、审计报表、数据保留策略和有效权限解释。

成功标准：配置变更、QA 诊断结论和权限判断都能被团队复核，并能形成可审计记录。

## 4. 取项原则

- 每个 Sprint 优先交付一条可验证主链路，不把多个复杂子系统塞进同一轮。
- P1 Backlog 优先于 P2；P2 只有在主链路完成且风险可控时进入实现。
- 每个 Sprint 至少保留一个脚本级验证入口，前端变更必须保留 `npm run build` 作为最小检查。
- 真实外部 Provider 失败不阻塞 local/mock 验收，但必须记录诊断结果和环境限制。
- 文档、接口、数据模型和实现如果出现冲突，以 PostgreSQL 业务真值、后端执行契约和可验证脚本为优先。

## 5. 风险与观察点

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 真实 Provider 环境不可用 | V1.2 生产化接入延迟 | 保留 local/mock 验收，真实环境用诊断报告单独复测 |
| EvaluationRun 范围膨胀 | Sprint 14 无法闭环 | 先做同步或轻量异步执行，不引入复杂调度平台 |
| 文档质量治理规则过多 | Sprint 16 变成规则平台 | 优先覆盖空 Chunk、重复 Chunk、解析失败和权限异常四类高价值问题 |
| 观测指标过细 | Sprint 17 维护成本升高 | 只采集 QARun、IngestJob、Provider 调用三类关键指标 |
| 协作治理过重 | Sprint 18 偏离内部工具定位 | 只做记录、评论、责任人和回滚确认，不做复杂审批流 |
