# 编码规范

## 1. 文档目的

本文档用于约束 RAG-Lab 后续开发阶段的命名、代码风格、注释、分层边界和 Git 提交流程。

本项目当前处于设计完成、准备进入开发阶段。编码规范应服务于小步快跑和可验证交付，不追求过度抽象，也不替代更具体的代码评审意见。

## 2. 基本原则

- 使用 UTF-8 编码保存所有代码、脚本、配置和文档。
- 优先沿用仓库已有目录、命名和组件风格。
- 只实现当前任务需要的最小可行范围，不提前设计无明确需求的扩展点。
- 代码、接口、数据库和文档必须保持同一套业务语义。
- 前端展示模型、后端 DTO、数据库字段和外部 Provider 返回值必须分层转换，不直接互相污染。
- 关键实现应有可执行验证；无法验证时，需要说明原因。

## 3. 命名规范

### 3.1 通用命名

| 对象 | 规则 | 示例 |
| --- | --- | --- |
| 业务概念 | 使用项目设计文档中的英文术语 | `KnowledgeBase`、`ConfigRevision`、`QARun` |
| 文件名 | 遵循所在技术栈和现有目录风格 | `pipelineService.ts`、`config_revision.py` |
| 目录名 | 使用小写，必要时使用短横线或下划线，保持同目录一致 | `components/rag`、`03-系统设计` |
| 常量 | 前端/后端代码内使用大写下划线 | `DEFAULT_PAGE_SIZE` |
| 环境变量 | 大写下划线，按组件前缀分组 | `POSTGRES_DSN`、`MILVUS_URI` |

### 3.2 前端命名

| 对象 | 规则 | 示例 |
| --- | --- | --- |
| React 组件 | `PascalCase` | `KnowledgeBaseOverview` |
| Hook | `use` + `PascalCase` | `useQARunStatus` |
| 普通变量 / 函数 | `camelCase` | `activeRevisionId`、`loadDocuments` |
| Type / Interface | `PascalCase` | `QARunTraceViewModel` |
| API 字段 | `camelCase` | `configRevisionId`、`createdAt` |
| CSS token | 保持设计系统变量命名 | `--terracotta`、`--border-warm` |

前端页面层不得直接散落 API 路径和后端 DTO 字段转换逻辑。正式研发时建议保留以下分层：

- `services/`：封装 API 调用。
- `adapters/`：后端 DTO 与页面 ViewModel 转换。
- `types/`：稳定业务类型。
- `components/`：只消费 props 和事件回调。

### 3.3 后端命名

| 对象 | 规则 | 示例 |
| --- | --- | --- |
| Python 模块 | `snake_case` | `qa_run_service.py` |
| 类名 | `PascalCase` | `QARunService` |
| 函数 / 变量 | `snake_case` | `create_qa_run` |
| Pydantic Schema | 按用途加后缀 | `QARunCreateRequest`、`QARunResponse` |
| Service | 业务对象 + `Service` | `ConfigRevisionService` |
| Provider | 能力对象 + `Provider` | `DenseRetrievalProvider` |

后端服务层应依赖 Provider 抽象，不直接把 Milvus、OpenSearch、Neo4j 或模型厂商 SDK 调用写入 QA 编排主流程。

### 3.4 数据库命名

| 对象 | 规则 | 示例 |
| --- | --- | --- |
| 表名 | 复数 `snake_case` | `knowledge_bases`、`qa_runs` |
| 字段名 | `snake_case` | `config_revision_id`、`created_at` |
| 主键 | `{entity}_id` | `kb_id`、`run_id` |
| 外键 | 引用对象主键名 | `document_id`、`version_id` |
| 唯一约束 | `uk_{table}_{columns}` | `uk_users_username_active` |
| 普通索引 | `idx_{table}_{columns}` | `idx_qa_runs_kb_status_created_at` |
| 部分唯一索引 | `uk_{table}_{condition}` | `uk_config_revisions_active_per_kb` |

数据库字段与 API 字段通过 DTO 或 Adapter 映射。数据库内部使用 `snake_case`，API 请求和响应使用 `camelCase`。

## 4. 代码风格

### 4.1 TypeScript / React

- 优先使用函数组件和显式 props 类型。
- 页面组件负责布局和交互编排，基础组件只负责展示。
- 复杂页面状态应抽到 Hook 或页面内局部 reducer，避免在 JSX 中写大量业务分支。
- 接口调用集中在 service 层，页面不直接拼接 URL。
- 不在组件中硬编码后端枚举解释；枚举展示应由 Adapter 或常量映射处理。
- UI 风格遵循根目录 `DESIGN.md` 和原型设计系统，不擅自改动整体视觉方向。

### 4.2 Python / FastAPI

- API 层只做请求解析、权限入口、响应组装，不承载复杂业务。
- Service 层承载业务流程和事务边界。
- Repository 或数据访问层承载数据库读写。
- Worker 任务只执行异步处理，不绕过 Service 层关键校验。
- Provider 实现负责外部组件适配，Provider 接口返回统一业务对象。
- 错误码、状态枚举和权限码应集中定义，避免散落字符串。

### 4.3 SQL / 迁移脚本

- 迁移脚本必须可重复审查，不在一个迁移中混入无关表或历史重构。
- 表结构、索引、默认值和约束应能追溯到 `数据模型设计.md` 或 `数据库设计.md`。
- 高风险迁移应说明回滚方式或补偿方式。
- 默认使用逻辑删除或归档策略，除非设计文档明确允许物理删除。

## 5. 注释规范

- 新增注释应解释“为什么这样做”，不要重复代码表面的“做了什么”。
- 关键函数或复杂流程需要函数级注释，说明输入、输出、约束和副作用。
- 安全边界、事务边界、权限裁剪、跨存储同步等逻辑必须有简短注释。
- 不删除看不懂但与当前任务无关的历史注释。
- 文档注释和代码实现不一致时，以设计文档和代码评审结论为准，并及时更新过期文档。

## 6. 分层边界

### 6.1 前端边界

- 页面层不直接依赖数据库字段名、模型厂商名或检索组件实现。
- `pipelineDefinition` 是业务契约，不等同于画布节点 ViewModel。
- P08 配置中心负责正式 Revision 的保存和激活；P09 QA 调试只做单次运行和临时覆盖。
- 权限按钮隐藏只是交互优化，不能替代后端鉴权。

### 6.2 后端边界

- QA 编排流程不得绕过 `permissionFilter`。
- Graph Retrieval 输出必须回落到授权 Chunk 后才能进入 Evidence 和 Citation。
- Config Revision 激活、文档 active version 切换等操作必须有事务和审计。
- 历史 QARun 指向执行时锁定的 `configRevisionId`，不随 active Revision 变化。

### 6.3 数据边界

- PostgreSQL 是业务主数据真值中心。
- Chunk 正文真值必须保存在 PostgreSQL。
- MinIO、Milvus、OpenSearch、Neo4j 不能替代 PostgreSQL 的业务真值。
- 检索副本可以重建，重建来源应是 PostgreSQL 和对象存储中的真值数据。

## 7. Git 规范

### 7.1 分支命名

| 类型 | 规则 | 示例 |
| --- | --- | --- |
| Codex 自动开发 | `codex/{short-topic}` | `codex/docs-dev-prep` |
| 功能开发 | `feature/{short-topic}` | `feature/qa-run-api` |
| 缺陷修复 | `fix/{short-topic}` | `fix/config-activation` |
| 文档更新 | `docs/{short-topic}` | `docs/database-design` |
| 重构 | `refactor/{short-topic}` | `refactor/provider-adapter` |

若用户或团队有更具体分支要求，以当次要求为准。

### 7.2 提交信息

提交信息推荐使用简洁中文或英文 Conventional Commits：

```text
docs: 补充数据库设计说明
feat: add qa run status polling
fix: 修复配置激活事务不一致
refactor: split pipeline adapter
test: add config revision validation cases
```

常用类型：

| 类型 | 说明 |
| --- | --- |
| `feat` | 新功能 |
| `fix` | 缺陷修复 |
| `docs` | 文档 |
| `test` | 测试 |
| `refactor` | 不改变行为的重构 |
| `chore` | 构建、依赖、脚手架等杂项 |

### 7.3 提交粒度

- 一次提交只表达一个清晰意图。
- 不把格式化、重命名和业务变更混在一起。
- 不提交本地环境文件、日志、构建产物和临时调试文件。
- 提交前至少检查 `git diff`，确认没有无关改动。

## 8. 验证要求

- 文档改动：检查链接、术语和现有设计文档一致。
- 前端改动：至少运行相关构建或页面级验证。
- 后端改动：至少运行相关单元测试、接口测试或最小复现。
- 数据库改动：检查迁移可执行性、索引和约束与查询场景一致。
- 无法执行验证时，在结果说明中写明缺失依赖、环境限制或阻塞原因。
