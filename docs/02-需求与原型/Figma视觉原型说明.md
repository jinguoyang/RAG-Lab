# RAG 调试平台 Figma 视觉原型说明

更新时间：2026-04-22

## 1. 文档目的

本文档用于将 [DESIGN.md](../../DESIGN.md) 的视觉语言转译为适合 RAG 调试平台后台系统的 Figma 原型设计规则，并作为 [原型设计文档.md](./原型设计文档.md) 的视觉补充说明。

本文档重点解决以下问题：

- Claude 风格中哪些元素适合后台系统保留
- 后台系统应采用怎样的统一版式和组件语言
- QA 调试页如何在高信息密度下仍保持秩序和美感
- Figma 中应如何组织画板、组件和页面状态

## 2. 视觉定位

本项目的视觉方向不是“Claude 官网风”，而是“Claude 气质的知识工作台后台系统”。

设计关键词：

- 温暖
- 克制
- 专业
- 可追溯
- 高信息密度
- 编辑感而非营销感

目标感受：

- 像在一张高质量纸面上处理知识、版本与证据
- 比传统蓝白后台更有温度
- 比内容型网站更强调工具秩序和操作效率

## 3. 设计转译原则

### 3.1 建议保留的 Claude 风格元素

- 全局暖羊皮纸背景：`#f5f4ed`
- 卡片/浮层象牙白表面：`#faf9f5`
- 主强调色 Terracotta：`#c96442`
- 暖中性色文本体系，避免冷灰
- 页面标题和关键模块标题使用 Serif，功能性 UI 统一 Sans
- `0px 0px 0px 1px` 的 warm ring shadow
- 轻分隔线、柔和圆角、弱阴影
- 页面标题区的编辑式节奏感

### 3.2 需要收敛的 Claude 风格元素

- 不使用官网式大篇幅留白
- 不使用营销 Hero 和大面积故事化编排
- 不使用有机插画作为后台主体视觉
- 不在表格、表单、筛选、长列表里使用 Serif
- 不使用 `24px / 32px` 的大圆角作为后台默认样式
- 不让 Terracotta 同时承担成功、失败等状态语义
- 不使用重阴影和强悬浮卡片堆叠

### 3.3 后台系统的平衡方式

- 保留纸感、色温和排版气质
- 提高栅格密度、信息密度和操作效率
- 用标题区表达气质，用工作区表达秩序

## 4. 颜色系统

## 4.1 基础色板

| 角色 | 颜色值 | 用途 |
| --- | --- | --- |
| Parchment | `#f5f4ed` | 全局页面背景 |
| Ivory | `#faf9f5` | 卡片、表格容器、抽屉背景 |
| Near Black | `#141413` | 主文本、深色标题 |
| Olive Gray | `#5e5d59` | 次级文本、说明文案 |
| Stone Gray | `#87867f` | 弱化信息、元数据 |
| Terracotta | `#c96442` | 主按钮、active revision、高价值动作 |
| Border Cream | `#f0eee6` | 轻边框、分隔线 |
| Border Warm | `#e8e6dc` | 更明显的边框与分区线 |
| Focus Blue | `#3898ec` | 焦点环，仅用于可访问性交互 |
| Error Red | `#b53333` | 错误态 |

## 4.2 状态色建议

| 状态 | 建议表达 |
| --- | --- |
| queued | 暖灰标签 + 图标 |
| running | Terracotta 边框/点缀 + 动态指示 |
| success | 独立绿色系，不与品牌色混用 |
| failed | Error Red |
| cancelled | Stone Gray |
| partial | 暖黄色或琥珀色提示条 |
| active | Terracotta 细边框 + 轻填充 |
| inactive | 暖灰轮廓标签 |
| filtered | 柔和灰黄提示标签 |

说明：

- `active revision`、`当前激活版本` 这类“当前选中态”使用 Terracotta。
- 成功/失败等系统状态不要直接复用品牌色。

## 5. 字体与层级

## 5.1 字体建议

- 标题：`Anthropic Serif` 或 Figma 中使用 `Georgia` 替代
- 正文/UI：`Anthropic Sans` 或 Figma 中使用干净无衬线替代
- Trace/代码：等宽字体

## 5.2 使用规则

- 页面标题使用 Serif
- 知识库名、QA Answer 标题、关键卡片标题可使用 Serif
- 所有按钮、表单、表格、标签、导航、抽屉正文全部使用 Sans
- Trace、RunId、DocId、ChunkId 等技术字段使用 Mono

## 5.3 建议字号

| 层级 | 字号 | 用途 |
| --- | --- | --- |
| Page Title | 32px | 页面标题 |
| Section Title | 24px | 主要模块标题 |
| Card Title | 20px | 卡片或面板标题 |
| Body | 16px | 正文与 UI 主文本 |
| Caption | 14px | 元数据、说明 |
| Label | 12px | 标签、辅助字段 |

## 6. 布局系统

## 6.1 全局骨架

统一采用三段式后台布局：

- 左侧导航：`240px`
- 顶部上下文栏：`56-64px`
- 主内容区：自适应

知识库层页面顶部需固定展示：

- 当前知识库名称
- `active config revision`
- 当前用户角色
- 当前页面主操作

## 6.2 栅格与内容宽度

- 主内容区建议使用 `12 列栅格`
- 推荐设计宽度：`1360-1440px`
- 页面内容不做营销式窄栏居中
- 表格页、三栏页、配置页优先使用宽画布

## 6.3 页面头结构

统一页面头：

- 左侧：面包屑 + 页面标题 + 一行说明
- 右侧：主操作按钮 + 次操作按钮 + 上下文标签

标题区下方可插入系统提示条：

- 权限受限
- 未保存修改
- 部分降级成功
- 作业失败

## 7. 统一组件规则

## 7.1 卡片

- 背景：`#faf9f5`
- 边框：`1px solid #f0eee6`
- 圆角：`12px`
- 阴影：极轻 whisper shadow 或 ring shadow

适用：

- 统计卡
- 说明卡
- QA 结果区分块
- 配置摘要

## 7.2 表格

适用页面：

- P03 用户管理
- P04 用户组管理
- P06 文档中心
- P10 QA 历史
- P12 成员与权限

规则：

- 行高 `44-48px`
- 表头 sticky
- 使用暖边框分隔，不做强斑马纹
- 表格上方统一为筛选工具栏 + 批量操作条
- 行点击打开详情抽屉

## 7.3 抽屉

宽度建议：

- `480px`
- `560px`
- `640px`

适用：

- 用户详情
- 用户组成员
- Chunk 详情
- QA 历史详情

## 7.4 详情侧栏

右侧 sticky inspector 统一宽度：`320-360px`

适用：

- 文档详情
- 配置中心
- QA 调试页

内容放置建议：

- 对象 ID
- 版本信息
- 权限摘要
- 最近操作
- 状态说明

不建议放置长正文。

## 7.5 时间线

统一用于：

- Ingest 作业
- Revision 历史
- QA Trace

节点字段建议：

- 状态图标
- 步骤名
- 耗时
- 操作者
- 时间
- 展开摘要

## 7.6 状态标签

统一要求：

- 同时使用颜色、图标、文字
- 不能只靠颜色区分状态
- 风格偏轻量，不做大面积纯色块

## 8. 核心页面设计重点

## 8.1 P01 登录页

定位：

- 允许比后台内页更有 Claude 气质

设计重点：

- 左侧品牌说明区使用 Serif 标题
- 右侧登录卡片使用 Ivory 表面
- 主按钮使用 Terracotta

## 8.2 P02-P04 平台层页面

设计重点：

- 统一平台层导航
- 列表页以工具性为主，不做过度装饰
- 用页面标题区和柔和色温表达品牌调性

## 8.3 P05 知识库概览

设计重点：

- 统计卡不超过 4 张一排
- 用最近动态和快捷操作代替大块装饰性仪表盘
- `active revision` 作为重要上下文突出展示

## 8.4 P06-P07 文档体系

设计重点：

- 文档中心强调筛选、上传、重解析、状态追踪
- 文档详情强调版本、Chunk、作业三者的可切换关系
- Chunk 列表和详情抽屉是关键体验点

## 8.5 P08 配置中心

设计重点：

- 采用三栏式 Pipeline Designer：左侧模板与节点库，中间阶段式 Pipeline 画布，右侧 Node Inspector 与 Validation
- 要能明显看出 current active revision
- 差异摘要和风险确认是页面重点
- Revision 历史可作为抽屉或侧栏辅助信息，不应替代 Pipeline 主编辑区

## 8.6 P09 QA 调试页

这是整套原型中最重要的页面。

推荐布局：

- 顶部固定运行条
- 左栏 `320px`：Query、检索开关、参数、运行阶段
- 中栏 `min 720px`：Answer + Tabs（Evidence / Citation / Retrieved Chunks）
- 右栏 `360px`：Trace + Metrics + Run Context

设计原则：

- 不把每块都做成厚卡片
- 用浅底分区、标题条和细分隔线组织信息
- 中间主阅读流必须是：`Answer -> Evidence -> Citation`
- 右侧诊断信息要常驻但不喧宾夺主

关键组件：

- `Evidence Row`
- `Citation Chip`
- `Trace Step`
- `Run Stage Banner`

关键状态稿：

- 初始态
- 运行中
- 成功
- 部分降级成功
- 权限受限

## 8.7 P10 QA 历史页

设计重点：

- 与 QA 调试页保持视觉一致
- 更偏记录管理和回放入口
- 详情抽屉中应能快速预览结果快照

## 8.8 P11 图检索分析页

设计重点：

- 不走霓虹科技图谱风
- 用知识分析台的方式展示实体、关系、社区和回落 Chunk
- 权限裁剪结果要有明确提示

## 8.9 P12 成员与权限页

设计重点：

- 左侧成员列表 + 右侧能力矩阵
- 权限边界要清晰，但语气不应过度警示化

## 9. 状态稿要求

每个关键页面至少应补齐以下状态：

- Loading
- Empty
- Error
- Permission Denied
- Partial Degraded Success

页面级特殊要求：

- QA 调试页：运行中不要只给整页 spinner，要展示阶段流转
- 文档中心：失败作业要显示影响范围和重试入口
- 配置中心：未保存修改离开时要提示

## 12. 待确认项

在高保真设计前，建议确认以下问题：

1. `P09` 右侧 Trace 栏是否常驻 sticky，还是允许折叠。
2. `P08` 的 Revision 历史采用抽屉、侧栏还是底部时间线承载。
3. `P11` 图检索分析页是否需要半可视化关系图，而不只是列表解释。
4. 登录页是否允许比后台其余页面更强一点的品牌气质表达。
