# RAG 调试平台需求规格说明书

更新时间：2026-04-22

## 1. 文档目的

本文档用于定义 RAG 调试平台的正式需求规格，作为产品、研发、测试、验收和 AI 编码助手的共同输入。

本文档重点回答以下问题：

- 系统必须解决哪些问题
- 系统包含哪些角色与业务范围
- 系统必须具备哪些功能
- 功能、数据、接口、安全和性能的验收边界是什么

本文档偏重“需求是什么”，不替代详细技术实现设计。

## 2. 适用范围

本文档适用于一个面向个人或小组内部使用的 RAG 调试平台。

适用场景：

- 内部知识库建设与维护
- 文档解析、切块与索引
- 基于真实文档的问答调试
- 检索链路调优
- 基于 Neo4j 的图检索增强
- 权限受控的证据和引用回溯

不适用场景：

- 面向公网终端用户的大规模聊天产品
- 复杂工单审批流平台
- 以 BI 分析和成本结算为主的平台

## 4. 术语定义

| 术语 | 说明 |
| --- | --- |
| KB | Knowledge Base，知识库 |
| Document | 知识库中的业务文档对象 |
| Document Version | 文档的可索引版本 |
| Chunk | 文档切分后的最小检索单元 |
| Dense Retrieval | 基于向量的语义检索 |
| Sparse Retrieval | 基于 BM25 等文本索引的关键词检索 |
| pgvector | PostgreSQL 的向量扩展，适用于中小规模向量检索 |
| 图检索增强 | 参考 GraphRAG 思路，通过 Neo4j 图结构做实体、关系和社区扩展检索 |
| Evidence | 回答生成时使用的证据片段 |
| Citation | 回答中最终暴露给用户的引用信息 |
| Config Revision | 某个知识库的一次完整配置版本 |

## 5. 产品目标

### 5.1 总目标

系统应提供一个围绕真实文档、真实检索链路和真实模型调用的 RAG 调试工作台，帮助研发和维护人员完成知识库管理、文档索引、问答调试、链路诊断与持续优化。

### 5.2 子目标

- 支持权限可控的知识库管理。
- 支持文档版本化和可追溯的 chunk 存储。
- 支持 Dense 检索，并在需要关键词检索或混合检索时支持 Sparse 检索。
- 支持参考 GraphRAG 思路、基于 Neo4j 的图检索增强。
- 支持问答调试、Trace、evidence、citation 和历史记录。
- 支持配置版本化，确保调试结果可追溯。

### 5.3 非目标

- 不要求实现多租户运营体系。
- 不要求实现复杂审批流。
- 不要求实现成本结算中心。
- 不要求以 mock 数据作为默认主链路。

## 6. 角色与用户

### 6.1 平台角色

- 平台管理员
- 平台普通用户

### 6.2 知识库角色

- 知识库负责人
- 知识库编辑者
- 问答操作员
- 只读用户

### 6.3 角色目标

| 角色 | 核心目标 |
| --- | --- |
| 平台管理员 | 管理用户、用户组、平台级配置和全局策略 |
| 知识库负责人 | 管理指定知识库、成员、文档、配置与调试结果 |
| 知识库编辑者 | 上传文档、重解析文档、修改配置、执行调试 |
| 问答操作员 | 执行问答调试、查看授权范围内结果 |
| 只读用户 | 查看授权范围内的知识库与调试摘要 |

## 7. 业务范围

### 7.1 In Scope

- 用户与用户组管理
- 知识库 CRUD 与成员绑定
- 模板管理与配置版本化
- 文档上传、版本化、重解析、chunk 查看
- Ingest 作业跟踪
- Dense / Sparse 检索
- Neo4j 图检索增强
- QA 调试
- QA 历史记录与回放
- 权限、密级与文档 ACL 控制

### 7.2 Out of Scope

- 外部客户租户隔离
- 复杂审批流
- 商业化计费
- 海量统计报表中心

## 8. 总体业务流程

1. 用户登录系统。
2. 用户进入某个知识库。
3. 用户上传文档并触发解析。
4. 系统生成文档版本、chunk 和必要的检索副本。
5. 系统根据文档内容抽取图结构并写入 Neo4j。
6. 用户配置或切换知识库的运行参数。
7. 用户执行 QA 调试。
8. 系统按权限过滤执行 Dense、Sparse 和图检索增强。
9. 系统生成回答并返回 evidence、citation、trace 与指标。
10. 用户回看历史调试结果并继续优化配置。

## 9. 功能需求

以下需求使用统一编号：

- `FR`：功能需求
- `DR`：数据需求
- `IR`：接口需求
- `SR`：安全与权限需求
- `NFR`：非功能需求

### 9.1 用户与组织管理

#### FR-001 用户登录

系统必须支持用户登录，并识别当前用户身份。

验收口径：

- 支持至少一种正式认证方式。
- 登录后可获取当前用户信息。
- 未登录用户不得访问受控接口。

#### FR-002 用户管理

系统必须支持平台管理员维护用户基础资料。

验收口径：

- 可新增、禁用、查看用户。
- 用户信息至少包含用户名、展示名、邮箱、平台角色、密级。

#### FR-003 用户组管理

系统必须支持平台管理员管理用户组和组成员。

验收口径：

- 可创建用户组。
- 可给用户组添加或移除成员。
- 用户组可用于知识库角色绑定与文档 ACL。

### 9.2 知识库管理

#### FR-010 知识库创建与维护

系统必须支持创建、编辑、查看和停用知识库。

验收口径：

- 知识库至少包含名称、描述、状态、默认密级、负责人、索引能力开关。
- 知识库名称在系统内可读、可检索。

#### FR-011 知识库成员绑定

系统必须支持按用户或用户组绑定知识库角色。

验收口径：

- 支持绑定 `kb_owner / kb_editor / kb_operator / kb_viewer`。
- 支持查看当前知识库成员和角色。

### 9.3 模板与配置管理

#### FR-020 模板管理

系统必须支持模板的创建、查看、修改和应用。

验收口径：

- 模板至少包含文档处理配置、Pipeline 配置、检索配置、生成配置。
- 模板可被应用到某个知识库。

#### FR-021 配置版本化

系统必须支持知识库配置版本化。

验收口径：

- 每次保存配置都会生成 revision。
- 系统可以识别当前 active revision。
- QA 运行记录可以追溯到配置版本。

### 9.4 文档管理

#### FR-030 文档上传

系统必须支持向知识库上传文档。

验收口径：

- 上传后生成稳定的文档对象。
- 上传后生成新的文档版本。
- 上传动作必须记录操作人。

#### FR-031 文档版本管理

系统必须支持文档版本化与活跃版本切换。

验收口径：

- 同一文档可存在多个版本。
- 任一时刻只允许一个 active version。
- 重解析生成新版本时，旧版本保留可追溯关系。

#### FR-032 文档重解析

系统必须支持对文档执行重解析和重建索引。

验收口径：

- 重解析后生成新的文档版本。
- 新版本可替换为 active version。
- 旧版本索引可延迟清理。

#### FR-033 Chunk 查看

系统必须支持查看文档 chunk 信息。

验收口径：

- 支持分页查看 chunk。
- 至少展示 chunk 正文、页码、章节、token 数、metadata。
- 查看 chunk 必须受权限控制。

### 9.5 Ingest 与索引

#### FR-040 Ingest 作业管理

系统必须支持 ingest 作业状态跟踪。

验收口径：

- 作业至少包含 `queued / running / success / failed / cancelled` 状态。
- 支持查看作业进度、错误信息和发起人。

#### FR-041 Dense 向量存储与检索

系统必须支持将 chunk embedding 写入 Dense 检索层。

验收口径：

- 优先支持写入 Milvus。
- 当数据规模不大或部署希望简化时，可直接使用 PostgreSQL `pgvector`。
- 向量写入和 chunk 主数据可建立回溯关系。
- Dense 检索结果必须可回表 PostgreSQL 中的 chunk 真值。

#### FR-042 Sparse 文本索引写入

当知识库或部署启用 Sparse 索引能力时，系统必须在文档入库、重解析和版本切换过程中将 chunk 文本索引写入 OpenSearch。

验收口径：

- OpenSearch 只作为关键词检索或混合检索的副本索引层。
- OpenSearch 中至少包含 chunk 基本文本和过滤字段。
- 可按知识库、文档、密级和版本进行过滤。
- 单次 QA 是否启用 Sparse 或 Hybrid 检索，不决定文档处理阶段是否写入 OpenSearch。
- Hybrid 是 QA 阶段同时使用 Dense 与 Sparse 并做融合的检索策略，不是单独的存储副本。

#### FR-043 图结构构建

当知识库或部署启用图索引能力时，系统必须在文档入库、重解析和版本切换过程中基于文档内容抽取实体、关系和社区，并写入 Neo4j。

验收口径：

- 可构建节点、关系和社区摘要。
- 图结构必须能回溯到支撑 chunk。
- 图结构构建状态可跟踪。
- 单次 QA 是否启用 Graph Retrieval，不决定文档处理阶段是否写入 Neo4j。

### 9.6 问答调试

#### FR-050 QA 调试执行

系统必须支持针对某个知识库执行单次问答调试。

验收口径：

- 输入至少包括 `kbId` 和 `query`。
- 输出至少包括 `answer`、`evidence`、`citations`、`trace`、`metrics`。

#### FR-051 Dense + Sparse 混合检索

系统必须支持 Dense 与 Sparse 两路召回。

验收口径：

- Dense 和 Sparse 均可独立开关。
- 候选结果可合并、去重和 rerank。

#### FR-052 Neo4j 图检索增强

系统必须支持参考 GraphRAG 思路，通过 Neo4j 做图检索增强。

验收口径：

- 可从 query 中识别图检索种子。
- 可做实体匹配、关系扩展、社区召回和支撑 chunk 反查。
- 图增强结果不能直接绕过权限和引用约束。

#### FR-053 引用与证据回溯

系统必须支持输出 evidence 和 citation。

验收口径：

- Evidence 必须能定位到 chunk。
- Citation 必须可展示来源文档和片段位置。
- 未授权 chunk 不得出现在 evidence 或 citation 中。

#### FR-054 Trace 与指标

系统必须支持返回 QA 调试过程 Trace 和核心指标。

验收口径：

- 至少记录步骤顺序、耗时、tokens、输入输出摘要。
- 至少返回总耗时、总 tokens、检索诊断结果。

#### FR-055 QA 历史记录

系统必须支持查看 QA 历史记录和明细。

验收口径：

- 支持按知识库分页查询。
- 支持查看单次运行明细。
- 默认仅发起人和高权限用户可查看完整结果。

## 10. 数据需求

### 10.1 数据真值

#### DR-001 PostgreSQL 真值要求

系统必须使用 PostgreSQL 保存业务主数据、配置、权限、运行记录和图构建元数据。

#### DR-002 对象存储真值要求

系统必须使用对象存储保存原始文件与大体积解析产物。

#### DR-003 Chunk 主存要求

系统必须优先使用 PostgreSQL 保存 chunk 正文和 chunk 级业务元数据。

验收口径：

- chunk 正文必须可与 `doc_id / document_version / section / page / token_count / metadata / permission / created_at` 等字段一起进行事务性写入和查询。
- PostgreSQL 必须作为 chunk 正文的主存储。
- 不得把 MinIO 作为 chunk 正文主存储。
- 不得把 Neo4j 作为 chunk 正文主存储。

#### DR-004 Dense 检索层要求

系统必须提供 Dense 向量存储能力。

验收口径：

- 优先支持 Milvus 作为向量检索层。
- 当规模不大时，可使用 PostgreSQL `pgvector` 作为替代方案。
- 无论使用哪种向量方案，chunk 正文真值仍必须留在 PostgreSQL。

#### DR-005 索引副本要求

Milvus 和 OpenSearch 只能作为检索副本，不得作为唯一真值来源。

#### DR-006 OpenSearch 启用边界

OpenSearch 只有在知识库或部署启用 Sparse 索引能力时才承担副本索引角色。单次 QA Run 是否使用 Sparse 或 Hybrid 只影响运行策略，不影响既有文档是否应写入 OpenSearch。

验收口径：

- 系统不应把 OpenSearch 作为 chunk 正文唯一来源。
- 在未启用 Sparse 索引能力的知识库或部署中，允许不启用 OpenSearch。

### 10.2 文档与 chunk 数据

#### DR-010 文档模型要求

系统必须区分 `document` 与 `document_version`。

#### DR-011 Chunk 真值要求

系统必须在 PostgreSQL 中保存 chunk 正文真值。

验收口径：

- chunk 正文不能只保存在向量库中。
- chunk 正文不能只保存在 OpenSearch 中。
- chunk 正文不能只保存在对象存储中。
- chunk 需要可追溯到文档版本。

#### DR-012 活跃版本要求

系统必须保证文档任一时刻只有一个 active version。

### 10.3 图数据

#### DR-020 Neo4j 图主存要求

系统必须使用 Neo4j 保存图结构主数据。

#### DR-021 图数据回溯要求

系统中所有实体、关系和社区都必须能追溯到支撑 chunk。

#### DR-022 图快照要求

系统必须记录图构建快照或构建作业元数据，以便审计和回滚。

## 11. 接口需求

### 11.1 通用接口要求

#### IR-001 资源标识

所有资源定位类参数必须统一使用 `...Id`。

#### IR-002 错误模型

所有 API 必须返回统一错误模型。

#### IR-003 鉴权要求

所有受控接口必须依赖后端鉴权。

### 11.2 核心接口组

必须提供以下接口域：

- Auth
- Users / Groups
- Knowledge Bases
- Templates / Config
- Documents / Ingest
- QA
- Graph

### 11.3 QA 接口最小返回要求

`QA Run` 接口至少必须返回：

- `runId`
- `status`
- `query`
- `retrievalDiagnostics`
- `evidence`
- `citations`
- `steps`
- `metrics`

## 12. 权限与安全需求

### 12.1 通用安全

#### SR-001 后端授权

系统必须在后端执行实际授权判断。

#### SR-002 默认拒绝

对于未明确授权的资源，系统默认拒绝访问。

#### SR-003 冲突策略

当 allow 与 deny 同时命中时，`deny` 必须优先。

### 12.2 文档权限

#### SR-010 文档读取

只有满足知识库角色、文档密级和 ACL 条件的用户才可读取文档。

#### SR-011 chunk 读取

只有具备 `kb.chunk.read` 的用户才可查看 chunk 正文。

#### SR-012 文档下载

只有具备 `kb.document.download` 的用户才可下载原始文档。

### 12.3 问答权限

#### SR-020 QA 执行权限

只有具备 `kb.qa.run` 的用户才可执行 QA 调试。

#### SR-021 检索阶段权限过滤

Dense、Sparse 和 Neo4j 图扩展都必须带上权限过滤。

#### SR-022 输出阶段权限裁剪

最终 answer 的 evidence、citation、trace 不得暴露未授权 chunk。

### 12.4 图检索安全

#### SR-030 图结构不等于证据

Neo4j 中的实体、关系和社区不能直接充当最终引用证据。

#### SR-031 图增强必须回落 chunk

图检索增强结果必须回落到用户有权访问的 chunk 才可进入回答上下文。

## 13. 非功能需求

### 13.1 可用性与可维护性

#### NFR-001 可追溯性

系统必须支持从 QA 结果追溯到：

- 知识库
- 配置版本
- 文档版本
- chunk

#### NFR-002 可维护性

系统应保持模块边界清晰，便于按模块迭代。

### 13.2 性能

#### NFR-010 调试响应目标

在正常研发环境下，单次 QA 调试应在可接受时间内返回。

说明：

- 本条需求不规定固定毫秒级 SLA。
- 具体目标可在实施阶段按环境补充。

#### NFR-011 检索可过滤

Dense 检索层必须支持按知识库、版本、密级和状态过滤；若启用 OpenSearch，则 OpenSearch 也必须支持相同过滤能力。

### 13.3 一致性

#### NFR-020 数据一致性

系统必须保证 PostgreSQL 主数据、对象存储产物、Dense 检索副本、可选的 OpenSearch 索引和 Neo4j 图结构之间具备可恢复的一致性关系。

#### NFR-021 重建能力

当向量索引、可选的 OpenSearch 索引或图结构丢失时，系统应支持基于 PostgreSQL 和对象存储重建。

## 14. 验收标准

### 14.1 MVP 验收

MVP 至少应满足：

- 可登录
- 可管理知识库
- 可上传文档
- 可生成 chunk
- 可将 chunk 正文写入 PostgreSQL
- 可写入 Milvus，或在中小规模场景下写入 pgvector
- 若启用 Sparse 索引能力，可写入 OpenSearch
- 可执行 QA 调试
- 可返回 evidence、citation、trace
- 可进行基础 Neo4j 图构建

### 14.2 权限验收

- 无权限用户不能访问未授权知识库。
- 无权限用户不能下载未授权文档。
- 无权限用户不能让未授权 chunk 参与问答输出。

### 14.3 图增强验收

- 图结构构建成功后，系统可执行图扩展查询。
- 图扩展查询结果最终能回落到授权 chunk。
- Neo4j 中的图结构不可直接替代 citation。

## 15. 与 AI 编码的关系

本文档可直接作为 AI 编码助手的需求输入。

建议搭配以下文档一起使用：

- 总设计主文档
- AI 开发设计包
- AI 启动提示词模板

建议不要直接对 AI 下达“实现整个系统”的任务，而应按需求条目或模块切片逐步派发。
