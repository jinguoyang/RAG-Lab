# 发布验收与运维手册

## 1. 本地环境部署

本地开发环境用于验证 V1.0 最小闭环，不直接作为生产运行环境。

1. 前端：进入 `frontend/` 后执行 `npm install`、`npm run build` 或 `npm run dev`。
2. 后端：进入 `backend/` 后执行 `conda env update -f environment.yml --prune`。
3. 配置：复制 `backend/.env.example` 为 `backend/.env`，至少配置 `RAG_LAB_DATABASE_URL`。
4. 数据库：执行 `backend/scripts/migrate.ps1` 应用 Alembic 迁移。
5. 启动：执行 `backend/scripts/start-dev.ps1`，访问 `/api/v1/health` 做基础探活。

## 2. 测试环境部署

测试环境沿用本地工程结构，但必须使用独立数据库和对象存储命名空间。

- `RAG_LAB_ENVIRONMENT` 设置为 `test`。
- 数据库连接使用测试专用 PostgreSQL，迁移完成后再导入种子数据。
- 外部 Provider 可先使用 `local` / `identity` 占位，接入真实 MinIO、Milvus、OpenSearch、Neo4j、LLM 前必须更新 `.env`。
- 发布前执行后端编译、OpenAPI 导出、Epic10 验证脚本和前端构建。

## 3. 外部依赖健康检查

后端提供 `GET /api/v1/health/dependencies` 返回依赖配置摘要。该接口只检查配置完整性，不主动发起外部网络探测，避免发布检查因 Provider 慢响应阻塞。

重点关注：

- `postgres` 为 `configured` 时表示已配置 `RAG_LAB_DATABASE_URL`。
- `object_storage` 为 `local` 表示仅保存元数据，不写真实对象存储。
- `embedding`、`dense_retrieval`、`sparse_retrieval`、`graph_retrieval`、`llm`、`rerank` 会返回当前 Provider 名称。

## 4. 备份恢复

PostgreSQL 是业务真值中心，备份恢复必须优先覆盖数据库。

- 备份：使用 `pg_dump` 导出业务库，并同步备份对象存储 bucket 中的原始文件。
- 恢复：先恢复 PostgreSQL，再恢复对象存储，最后按需重建 Milvus、OpenSearch 和 Neo4j 副本。
- 校验：恢复后执行迁移状态检查、`/api/v1/health/dependencies`、文档详情、QA 历史和图检索分析的抽样读取。
- 回滚：若新版本迁移失败，停止服务，恢复上一份数据库备份和对象存储快照，再启动上一版本镜像或代码包。
- 备份恢复演练记录：演练完成后通过 `POST /api/v1/knowledge-bases/{kbId}/backup-drills` 回填结果，列表通过 `GET /api/v1/knowledge-bases/{kbId}/backup-drills` 查询；记录必须包含 `restoredObjects`、`residualRisks`、演练结果和可追溯证据。

## 6. 审计排障

高风险操作通过 `GET /api/v1/audit-logs` 查询，支持按 `kbId`、`action`、`resourceType`、`resourceId`、`actorId`、`createdFrom`、`createdTo` 和 `keyword` 过滤。

非平台管理员必须指定 `kbId`，且只能查询自己具备 `kb.view` 权限的知识库日志。
