# 数据库设计

## 1. 文档目的

本文档面向后续数据库迁移脚本、后端 Repository、接口联调和测试验收，整理 RAG-Lab 数据库落地设计。

详细字段定义、跨存储关系和状态说明以 [数据模型设计](./数据模型设计.md) 为来源。本文档不提供可直接执行的 SQL，也不替代后续 Alembic 或其他迁移脚本。

## 2. 设计原则

- PostgreSQL 是业务主数据真值中心。
- 表名、字段名、索引名和约束名统一使用 `snake_case`。
- API 层使用 `camelCase`，由 DTO 或 Adapter 负责与数据库字段映射。
- 主键默认使用 UUID，字段名采用 `{entity}_id`。
- 所有外键字段必须显式命名，不通过业务编码隐式关联。
- Chunk 正文真值必须保存在 PostgreSQL，不能只保存在 MinIO、Milvus、OpenSearch 或 Neo4j。
- Milvus、OpenSearch 和 Neo4j 均视为可重建副本或图结构主存，业务状态仍以 PostgreSQL 为准。

## 3. 通用字段

业务主表默认包含审计字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `created_at` | `timestamptz` | 创建时间 |
| `created_by` | `uuid` | 创建人，系统任务可为空 |
| `updated_at` | `timestamptz` | 最后修改时间 |
| `updated_by` | `uuid` | 最后修改人，系统任务可为空 |

支持逻辑删除的表增加：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `deleted_at` | `timestamptz` | 删除时间，为空表示未删除 |
| `deleted_by` | `uuid` | 删除人 |

历史事实表和作业表以追加记录和状态更新为主，通常不做逻辑删除，例如 `qa_runs`、`qa_run_trace_steps`、`ingest_jobs`、`audit_logs`。

## 4. 表结构说明

### 4.1 系统治理

| 表名 | 主键 | 说明 |
| --- | --- | --- |
| `system_dict_types` | `dict_type_id` | 字典类型，例如密级、来源类型、反馈标签 |
| `system_dict_items` | `dict_item_id` | 字典项，归属于字典类型 |
| `audit_logs` | `audit_log_id` | 关键业务操作审计，不替代业务表字段 |

落地要求：

- `system_dict_types.code` 在未删除数据中唯一。
- `system_dict_items` 在同一 `dict_type_id` 下 `code` 未删除唯一。
- 审计日志只追加，不逻辑删除，按审计保留策略归档。

### 4.2 用户与权限

| 表名 | 主键 | 说明 |
| --- | --- | --- |
| `users` | `user_id` | 系统用户主体 |
| `user_groups` | `group_id` | 用户组 |
| `user_group_members` | `group_member_id` | 用户与用户组关系 |
| `permissions` | `permission_id` | 稳定权限码定义 |
| `role_permission_bindings` | `role_permission_id` | 平台角色、知识库角色到权限码的默认映射 |
| `kb_member_bindings` | `binding_id` | 用户或用户组在知识库下的角色绑定 |
| `acl_rules` | `acl_rule_id` | 文档、版本或 Chunk 级授权规则 |
| `chunk_access_filters` | `access_filter_id` | 检索阶段访问过滤摘要 |

落地要求：

- 用户默认禁用或逻辑删除，不物理删除，避免破坏历史引用。
- 权限判断遵循默认拒绝和 deny 优先。
- `chunk_access_filters` 只用于减少越权召回范围，不替代最终授权判断。
- 用户组、角色、ACL 或密级变化后，需要刷新受影响 Chunk 的访问过滤摘要。

### 4.3 知识库

| 表名 | 主键 | 说明 |
| --- | --- | --- |
| `knowledge_bases` | `kb_id` | 知识库工作空间，关联文档、配置、成员、QA Run |

落地要求：

- `active_config_revision_id` 必须指向同一知识库下状态为 `active` 的 Revision。
- `sparse_index_enabled` 和 `graph_index_enabled` 是文档处理阶段的索引能力开关。
- `sparse_required_for_activation` 和 `graph_required_for_activation` 决定副本是否阻塞 active version 切换。

### 4.4 文件、文档与 Chunk

| 表名 | 主键 | 说明 |
| --- | --- | --- |
| `stored_files` | `file_id` | MinIO 对象引用和校验信息 |
| `documents` | `document_id` | 文档主对象 |
| `document_versions` | `version_id` | 文档版本 |
| `chunks` | `chunk_id` | 文档切块正文真值 |

落地要求：

- 同一 `document_id` 只允许一个 `document_versions.status = 'active'`。
- `document_versions.document_id + version_no` 唯一。
- `chunks.version_id + chunk_index` 唯一。
- 文档逻辑删除或版本切换后，由异步作业同步检索副本状态。
- `chunks.content` 是 QA Evidence 和 Citation 的最终回溯正文来源。

### 4.5 Ingest 与索引同步

| 表名 | 主键 | 说明 |
| --- | --- | --- |
| `ingest_jobs` | `job_id` | 文档解析、切块、索引和图构建作业 |
| `index_sync_jobs` | `sync_job_id` | 向 Milvus、OpenSearch、Neo4j 同步或重建副本的批次任务 |
| `index_sync_records` | `sync_record_id` | 单个 Chunk 或图对象的副本同步结果 |

落地要求：

- 重试作业必须创建新记录，不覆盖历史失败记录。
- `required_for_activation` 用于标记同步任务是否阻塞版本激活。
- 副本删除、重建和补偿均由 PostgreSQL 状态驱动。

### 4.6 配置

| 表名 | 主键 | 说明 |
| --- | --- | --- |
| `config_templates` | `template_id` | 可复用配置模板 |
| `config_revisions` | `config_revision_id` | 知识库配置版本 |

落地要求：

- `config_revisions.kb_id + revision_no` 唯一。
- 同一 `kb_id` 只允许一个 `status = 'active'` 的 Revision。
- Revision 激活必须在同一事务中更新旧 active、新 active、`knowledge_bases.active_config_revision_id` 和审计日志。
- `pipeline_definition` 保存后端执行契约，不保存画布坐标、颜色、图标等纯 UI 字段。

### 4.7 QA 调试与历史

| 表名 | 主键 | 说明 |
| --- | --- | --- |
| `qa_runs` | `run_id` | 单次 QA 调试记录 |
| `qa_run_trace_steps` | `trace_step_id` | Executed Pipeline Trace 步骤 |
| `qa_run_candidates` | `candidate_id` | 多路召回候选和裁剪原因 |
| `qa_run_evidence` | `evidence_id` | 授权 Evidence |
| `qa_run_citations` | `citation_id` | Citation 引用 |
| `evaluation_samples` | `sample_id` | 从历史运行沉淀的回归样本 |

落地要求：

- `qa_runs` 必须记录执行时锁定的 `config_revision_id`。
- 临时覆盖参数写入 `override_snapshot`，不得修改 active Revision。
- Evidence 只能引用授权后的 `chunks.chunk_id`。
- 历史 Evidence 快照读取时仍需按当前权限二次校验。
- Trace 中不得保存未授权 Chunk 正文。

### 4.8 图检索元数据

| 表名 | 主键 | 说明 |
| --- | --- | --- |
| `graph_snapshots` | `graph_snapshot_id` | 图构建快照元数据 |
| `graph_chunk_refs` | `graph_chunk_ref_id` | Neo4j 图对象到 PostgreSQL Chunk 的回溯摘要 |

落地要求：

- Neo4j 保存图结构，PostgreSQL 保存快照元数据和回表关系。
- 图检索结果必须通过 `chunk_id` 做最终权限裁剪。
- 文档 active version 切换、Chunk 删除、ACL 或密级变化时，受影响图快照必须标记为 `stale`。

## 5. 索引设计

### 5.1 唯一约束

| 约束 | 说明 |
| --- | --- |
| `users.username` 未删除唯一 | 避免登录名冲突 |
| `system_dict_types.code` 未删除唯一 | 避免字典类型编码冲突 |
| `system_dict_items.dict_type_id + code` 未删除唯一 | 避免同类型字典项编码冲突 |
| `document_versions.document_id + version_no` 唯一 | 保证文档内版本号稳定 |
| `config_revisions.kb_id + revision_no` 唯一 | 保证知识库内配置版本号稳定 |
| `chunks.version_id + chunk_index` 唯一 | 保证版本内 Chunk 顺序稳定 |

### 5.2 部分唯一索引

| 索引 | 说明 |
| --- | --- |
| `document_versions(document_id) where status = 'active'` | 同一文档只允许一个 active version |
| `config_revisions(kb_id) where status = 'active'` | 同一知识库只允许一个 active Revision |

### 5.3 查询索引

| 索引字段 | 支撑场景 |
| --- | --- |
| `knowledge_bases(owner_id, status)` | 平台工作台和负责人筛选 |
| `documents(kb_id, status, updated_at)` | 文档中心列表 |
| `document_versions(document_id, status, created_at)` | 文档详情版本列表 |
| `chunks(kb_id, version_id, status, chunk_index)` | Chunk 分页、检索回表 |
| `ingest_jobs(kb_id, status, created_at)` | 作业列表和概览 |
| `index_sync_jobs(kb_id, target_store, status, created_at)` | 副本同步监控 |
| `qa_runs(kb_id, status, created_at)` | QA 历史列表 |
| `qa_runs(created_by, created_at)` | 个人历史查询 |
| `qa_run_candidates(run_id, source_type, rank_no)` | 运行详情候选展示 |
| `qa_run_evidence(run_id, evidence_order)` | Evidence 顺序展示 |
| `acl_rules(resource_type, resource_id, status)` | 资源授权查询 |
| `acl_rules(subject_type, subject_id, permission_code, status)` | 主体权限查询 |
| `audit_logs(resource_type, resource_id, created_at)` | 审计回溯 |

### 5.4 JSONB 索引

以下字段按实际查询需求增加 GIN 索引，不在首版无条件创建：

| 字段 | 说明 |
| --- | --- |
| `chunks.metadata` | Chunk 标签、页码、解析来源等扩展查询 |
| `config_revisions.pipeline_definition` | 配置诊断或节点级检索 |
| `qa_runs.metrics` | 运行指标分析 |
| `qa_run_trace_steps.metrics` | 步骤耗时和 token 分析 |
| `index_sync_jobs.scope` | 按同步范围定位作业 |

## 6. 数据字典

### 6.1 建议使用 CHECK 的稳定状态

| 字段 | 候选值 |
| --- | --- |
| `users.status` | `active`、`disabled` |
| `knowledge_bases.status` | `draft`、`active`、`disabled`、`archived` |
| `documents.status` | `active`、`disabled`、`archived` |
| `document_versions.status` | `processing`、`active`、`inactive`、`failed` |
| `document_versions.parse_status` | `pending`、`running`、`success`、`failed` |
| `document_versions.*_index_status` | `not_required`、`pending`、`running`、`success`、`failed` |
| `ingest_jobs.status` | `queued`、`running`、`success`、`failed`、`cancelled` |
| `index_sync_jobs.status` | `queued`、`running`、`success`、`failed`、`cancelled` |
| `config_revisions.status` | `draft`、`saved`、`active`、`archived`、`invalid` |
| `qa_runs.status` | `draft`、`queued`、`running`、`success`、`partial`、`failed`、`cancelled` |
| `graph_snapshots.status` | `queued`、`running`、`success`、`failed`、`stale` |

### 6.2 建议进入系统字典的运营项

| 字典类型 | 说明 |
| --- | --- |
| `security_level` | 用户、文档、Chunk 密级 |
| `document_source_type` | 上传、同步、导入等文档来源 |
| `file_role` | `source`、`parsed_artifact`、`attachment` 等文件角色 |
| `kb_role` | `kb_owner`、`kb_editor`、`kb_operator`、`kb_viewer` |
| `platform_role` | `platform_admin`、`platform_user` 的展示名称 |
| `feedback_status` | `unrated`、`correct`、`partially_correct`、`wrong`、`citation_error`、`no_evidence` |

权限码不建议作为运营字典随意新增或删除，应通过 `permissions` 和 `role_permission_bindings` 作为稳定后端契约维护。

## 7. 删除与保留策略

| 类型 | 表 | 策略 |
| --- | --- | --- |
| 系统配置 | `system_dict_types`、`system_dict_items` | 逻辑删除 |
| 主业务对象 | `users`、`user_groups`、`knowledge_bases`、`documents`、`config_templates`、`config_revisions` | 逻辑删除或归档 |
| 授权关系 | `user_group_members`、`kb_member_bindings`、`acl_rules` | 建议逻辑失效，并记录审计 |
| 文档版本与 Chunk | `document_versions`、`chunks` | 不物理删除，按状态和保留策略归档 |
| 历史事实 | `qa_runs`、`qa_run_trace_steps`、`qa_run_candidates`、`qa_run_evidence`、`qa_run_citations`、`ingest_jobs`、`audit_logs` | 只追加或归档 |
| 外部副本 | Milvus、OpenSearch、Neo4j | 由 PostgreSQL 状态驱动异步清理或重建 |

## 8. 验证关注点

- 表结构能覆盖 User、Group、KB、Document、Version、Chunk、Ingest、ConfigRevision、QARun、Evidence、Citation、Trace、GraphSnapshot、EvaluationSample。
- 索引能支撑文档中心、QA 历史、作业列表、权限裁剪和检索回表。
- 状态枚举与接口设计、详细设计保持一致。
- PostgreSQL、MinIO、Milvus、OpenSearch、Neo4j 的数据职责没有互相替代。
- 权限过滤摘要只用于召回前过滤，最终输出仍必须回表 PostgreSQL 校验。
