# 接口设计说明

## 1. 文档目的

本文档定义 RAG 调试平台后端 API 的设计约定、接口清单、关键接口入参与出参、鉴权规则、错误模型和调用约束。

本文档面向前端联调、后端开发、测试验收和后续 OpenAPI 细化使用。本文档为接口草案级设计，不替代最终代码中的 OpenAPI Schema、字段校验实现和权限矩阵配置。

### 2.2 覆盖范围

本文覆盖以下接口域：

- Auth：登录、退出、当前用户
- Users / Groups：用户、用户组与组成员
- Knowledge Bases：知识库、概览、成员绑定、权限摘要
- Templates / Config Revisions：模板、Pipeline 校验、配置保存、激活、Diff
- Documents / Versions / Chunks：文档上传、版本、Chunk、重解析
- Ingest Jobs：作业查询、重试、取消
- QA Runs / QA History：单次调试、运行详情、历史查询、反馈、回放
- Graph / GraphSnapshots：图快照、实体搜索、关系路径、支撑 Chunk
- Evaluations：回归样本沉淀

### 2.3 非覆盖范围

- 不定义模型厂商、Milvus、OpenSearch、Neo4j 的底层 SDK 接口。
- 不定义数据库 DDL 和迁移脚本。
- 不定义完整 OpenAPI JSON / YAML 文件。
- 不定义公网终端聊天 API。

## 3. 通用接口约定

### 3.1 协议与路径

| 项 | 约定 |
| --- | --- |
| 协议 | HTTPS，开发环境可使用 HTTP |
| 风格 | RESTful API |
| 基础路径 | `/api/v1` |
| 数据格式 | `application/json; charset=utf-8` |
| 文件上传 | `multipart/form-data` |
| 时间格式 | ISO 8601，例如 `2026-04-24T10:30:00+08:00` |
| ID 格式 | API 层统一使用 `...Id`，值为 UUID 字符串 |
| 字段命名 | 请求和响应使用 `camelCase`；数据库内部可使用 `snake_case` |

### 3.2 鉴权方式

- 登录成功后由后端签发访问令牌，后续请求通过 `Authorization: Bearer <token>` 传递。
- 所有受控接口必须执行后端鉴权，前端按钮隐藏或置灰只作为交互优化。
- 未登录访问返回 `AUTH_UNAUTHORIZED`。
- 已登录但无权限访问返回 `AUTH_FORBIDDEN`，或在列表场景中直接过滤不可见资源。
- 权限判断遵循默认拒绝、Deny 优先、检索阶段和输出阶段双重裁剪原则。

### 3.3 请求头

| 请求头 | 必填 | 说明 |
| --- | --- | --- |
| `Authorization` | 受控接口必填 | Bearer Token |
| `Content-Type` | 是 | JSON 请求使用 `application/json; charset=utf-8`，文件上传使用 `multipart/form-data` |
| `X-Request-Id` | 否 | 前端生成的请求 ID，便于幂等和排障 |
| `X-Trace-Id` | 否 | 链路追踪 ID；未传时由后端生成 |

### 3.4 统一响应格式

成功响应统一使用以下结构：

```json
{
  "success": true,
  "data": {},
  "traceId": "trace-20260424-0001"
}
```

失败响应统一使用以下结构：

```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "请求参数不合法",
    "detail": "pipelineDefinition 至少需要启用一路检索",
    "fields": [
      {
        "field": "pipelineDefinition.nodes",
        "message": "Dense、Sparse、Graph 至少启用一路"
      }
    ]
  },
  "traceId": "trace-20260424-0002"
}
```

说明：

- `message` 面向前端展示，应稳定、简洁。
- `detail` 面向排障，可按环境控制是否返回。
- `fields` 用于表单或 Pipeline 校验的字段级错误。
- 响应中的 `traceId` 必须可关联后端日志。

### 3.5 分页、排序与筛选

分页查询统一使用 Query 参数：

| 参数 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `pageNo` | integer | 否 | 1 | 页码，从 1 开始 |
| `pageSize` | integer | 否 | 20 | 每页数量，建议最大 100 |
| `sortBy` | string | 否 | `createdAt` | 排序字段 |
| `sortOrder` | string | 否 | `desc` | `asc` / `desc` |
| `keyword` | string | 否 | 空 | 关键词搜索 |

分页响应统一结构：

```json
{
  "items": [],
  "pageNo": 1,
  "pageSize": 20,
  "total": 0
}
```

### 3.6 幂等与并发控制

- 文件上传、创建 QARun、重试作业等可能重复提交的接口，应支持 `X-Request-Id` 或业务幂等键。
- Revision 激活、文档 active version 切换等高风险操作应记录审计日志。
- 对需要防止覆盖的更新接口，可在后续实现中使用 `updatedAt` 或 `version` 做乐观并发控制。

### 3.7 接口稳定性与组件解耦原则

- 接口字段应面向业务语义设计，不面向某个前端 UI 组件结构设计。
- 后端 DTO 与前端页面 ViewModel 必须允许分离，前端通过 Adapter 转换后再传给页面组件。
- 接口不得暴露具体 UI 组件名称，例如 Card、Drawer、Canvas、NodeCard 等。
- `pipelineDefinition` 是业务契约，应描述阶段、节点类型、参数、约束和 Provider 能力，不应描述画布坐标、颜色、图标等纯展示信息。
- QA Run 接口应返回可转换为 Trace、Candidate、Evidence、Citation、Metrics 的稳定业务数据，不应绑定某个调试页面布局。
- 检索、模型、图谱等后端组件替换时，应尽量保持接口业务语义稳定，只在 Provider / Adapter 层处理实现差异。

## 6. 关键接口详情

### 6.1 登录

- 路径：`POST /api/v1/auth/login`
- 描述：校验用户凭据并返回访问令牌。
- 权限要求：无。

#### 请求参数

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `username` | string | 是 | 用户名或邮箱 |
| `password` | string | 是 | 密码 |

#### 响应字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `accessToken` | string | 访问令牌 |
| `expiresIn` | integer | 有效期，单位秒 |
| `user` | `UserDTO` | 当前用户摘要 |

#### 异常码

| 错误码 | 说明 | 处理建议 |
| --- | --- | --- |
| `AUTH_INVALID_CREDENTIALS` | 用户名或密码错误 | 前端提示认证失败，不暴露具体原因 |
| `AUTH_USER_DISABLED` | 用户已禁用 | 提示联系管理员 |

### 6.2 当前用户

- 路径：`GET /api/v1/auth/me`
- 描述：获取当前登录用户信息、平台角色和可见能力摘要。
- 权限要求：登录用户。

#### 响应字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `user` | `UserDTO` | 当前用户 |
| `platformPermissions` | string[] | 平台级权限码 |
| `visibleKbCount` | integer | 当前用户可见知识库数量 |

### 6.3 创建知识库

- 路径：`POST /api/v1/knowledge-bases`
- 描述：创建知识库，并自动为负责人创建 `kb_owner` 绑定。
- 权限要求：平台管理员或具备知识库创建权限的用户。

#### 请求参数

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `name` | string | 是 | 知识库名称 |
| `description` | string | 否 | 描述 |
| `ownerId` | string | 是 | 负责人用户 ID |
| `defaultSecurityLevel` | string | 是 | 默认密级 |
| `sparseIndexEnabled` | boolean | 否 | 是否维护 OpenSearch 文本索引副本，默认按部署策略 |
| `graphIndexEnabled` | boolean | 否 | 是否维护 Neo4j 图索引副本，默认按部署策略 |

#### 响应字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `kbId` | string | 知识库 ID |
| `status` | string | 初始状态，默认 `active` |
| `sparseIndexEnabled` | boolean | 实际生效的 Sparse 索引能力 |
| `graphIndexEnabled` | boolean | 实际生效的图索引能力 |
| `requiredForActivation` | object | 实际生效的激活门槛 |
| `ownerBindingId` | string | 自动生成的负责人绑定 ID |

#### 异常码

| 错误码 | 说明 | 处理建议 |

| `VALIDATION_ERROR` | 字段校验失败 | 高亮表单字段 |

### 6.4 上传文档

- 路径：`POST /api/v1/knowledge-bases/{kbId}/documents`
- 方法：`multipart/form-data`
- 描述：上传原始文件，创建 `Document`、`DocumentVersion` 和 `IngestJob`。
- 权限要求：`kb.document.upload`。

#### 请求参数

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `file` | file | 是 | 原始文档文件 |
| `name` | string | 否 | 文档名称，缺省使用文件名 |
| `securityLevel` | string | 否 | 文档密级，缺省使用知识库默认密级 |
| `metadata` | object | 否 | 来源、标签等扩展信息 |

#### 响应字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `document` | `DocumentDTO` | 文档对象 |
| `version` | `DocumentVersionDTO` | 新建文档版本 |
| `ingestJob` | `IngestJobDTO` | 入库作业 |

#### 设计约束

- 上传成功后即使后续解析失败，也应保留 `DocumentVersion` 与 `IngestJob`，用于前端展示失败原因和重试入口。
- Chunk 正文必须由后续解析任务写入 PostgreSQL，不得只保存在对象存储或检索副本中。
- 同一请求重复提交时，应尽量通过 `X-Request-Id` 避免重复创建。

| `STORAGE_WRITE_FAILED` | 对象存储写入失败 | 提示稍后重试并记录告警 |

### 6.5 切换文档 active version

- 路径：`POST /api/v1/knowledge-bases/{kbId}/documents/{documentId}/versions/{versionId}/activate`
- 描述：将指定版本设为文档 active version。
- 权限要求：`kb.document.upload`。

#### 请求参数

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `confirmImpact` | boolean | 是 | 前端二次确认标记，必须为 `true` |
| `reason` | string | 否 | 切换原因 |

#### 响应字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `documentId` | string | 文档 ID |
| `activeVersionId` | string | 新 active version ID |
| `previousActiveVersionId` | string | 原 active version ID |
| `auditLogId` | string | 审计日志 ID |

#### 设计约束

- `failed` 状态版本不得激活。
- 目标版本必须满足知识库索引能力配置的最小检索就绪门槛：解析成功，标记为 `requiredForActivation` 的 Dense / Sparse / Graph 副本状态为 `success`，未启用能力为 `not_required`。
- Config Revision 或本次 QARun 的 Sparse / Graph 开关只决定运行时是否调用对应检索 Provider，不决定文档版本激活前是否应维护 OpenSearch 或 Neo4j 副本。
- 切换后不删除旧版本和旧 Chunk，历史 QA 仍可追溯旧版本。
- 后续检索副本清理或重建通过异步作业处理。
- 切换后若旧图快照不再匹配当前 active version，必须标记为 `stale`，后续 QA 默认不使用该快照。

### 6.6 Pipeline 校验

- 路径：`POST /api/v1/knowledge-bases/{kbId}/config-revisions/validate`
- 描述：基于 `pipelineDefinition` 和 Provider 能力声明执行后端校验。
- 权限要求：`kb.config.manage`。

#### 请求参数

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `pipelineDefinition` | object | 是 | 受约束 Pipeline 定义 |
| `baseRevisionId` | string | 否 | 基线 Revision，用于差异提示 |

#### 响应字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `valid` | boolean | 是否通过校验 |
| `errors` | array | 阻断性错误 |
| `warnings` | array | 非阻断性提示 |
| `normalizedPipelineDefinition` | object | 后端规范化后的定义，可用于保存 |

#### 最小校验规则

| 规则 | 失败错误码 | 说明 |
| --- | --- | --- |
| 至少启用 Dense、Sparse、Graph 中一路 | `PIPELINE_NO_RETRIEVAL_ENABLED` | 全关闭时不可保存和运行 |
| `permissionFilter` 必须存在且在生成前 | `PIPELINE_PERMISSION_FILTER_REQUIRED` | 安全护栏不可绕过 |
| `citation` 必须来自授权 Evidence | `PIPELINE_CITATION_INVALID` | Citation 不得引用未授权内容 |
| Query Rewrite 若启用必须在 Retrieval 前 | `PIPELINE_STAGE_ORDER_INVALID` | 阶段顺序固定 |
| Graph Retrieval 输出必须回落 Chunk | `PIPELINE_GRAPH_CHUNK_FALLBACK_REQUIRED` | 图结构不能直接作为证据 |

#### 请求参数

### 6.8 激活 Config Revision

- 路径：`POST /api/v1/knowledge-bases/{kbId}/config-revisions/{revisionId}/activate`
- 描述：将指定 Revision 设为知识库 active revision。
- 权限要求：`kb.config.manage`。

#### 请求参数

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `confirmImpact` | boolean | 是 | 二次确认标记，必须为 `true` |
| `reason` | string | 否 | 激活原因 |

#### 响应字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `activeConfigRevisionId` | string | 新 active Revision ID |
| `previousActiveConfigRevisionId` | string | 原 active Revision ID |
| `activatedAt` | string | 激活时间 |
| `auditLogId` | string | 审计日志 ID |

#### 设计约束

- `invalid`、未通过校验或不属于当前知识库的 Revision 不允许激活。

- 路径：`POST /api/v1/knowledge-bases/{kbId}/qa-runs`

| `overrideParams` | object | 否 | 仅本次有效的临时覆盖参数 |
| `sourceRunId` | string | 否 | 回放来源 run ID |
| `remark` | string | 否 | 实验备注 |

#### 响应字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `runId` | string | QA Run ID |
| `status` | string | 创建后默认为 `queued` |
| `kbId` | string | 知识库 ID |
| `configRevisionId` | string | 本次锁定的配置版本 |
| `query` | string | 原始问题 |
| `createdAt` | string | 创建时间 |

#### 设计约束

- 创建时必须锁定本次使用的 `configRevisionId`。
- 临时覆盖参数不得修改 active revision，必须写入 `overrideSnapshot`。
- Dense、Sparse、Graph 至少启用一路，否则拒绝运行。
- Graph 结果必须回落到授权 Chunk / Evidence 后才能进入上下文。

- 部分检索通道失败但仍能生成答案时，状态应为 `partial`，并在 trace 中说明降级原因。

权限码来源说明：

- 平台角色和知识库角色必须通过后端 `permissions` 与 `rolePermissionBindings` 解析为权限码。
- 文档、版本和 Chunk 级 ACL 用于补充或覆盖角色权限，冲突时 deny 优先。
- 前端只能消费权限摘要，不允许自行推导最终授权结果。

## 9. 安全与权限约束

### 9.1 列表接口可见性

- 知识库列表只返回当前用户可见的知识库。
- 文档列表只返回当前用户具备 `kb.document.read` 权限的文档摘要。
- QA 历史列表默认返回发起人自己的记录；具备 `kb.qa.history.read` 的用户可查看知识库范围内记录。
- Dense、Sparse、Graph 检索请求中的权限过滤条件必须由后端根据角色、用户组、ACL、密级和 `ChunkAccessFilter` 生成，不接受前端传入的主体级过滤条件。
- 当检索 Provider 不支持主体级过滤时，后端必须先计算授权 Chunk 范围，再限制召回范围。

### 9.2 Chunk 与 Citation 安全

- Chunk 正文读取必须具备 `kb.chunk.read`。
- QA Evidence 和 Citation 只能基于授权 Chunk 构建。
- 被权限裁剪的候选只能以数量、状态或裁剪原因进入诊断摘要，不得暴露正文。
- 图检索命中的实体、关系和社区不得直接作为 Citation。
- 历史 Evidence 快照读取必须按当前用户权限二次校验；无权限时只能返回来源摘要、hash、脱敏状态或“当前无权查看”提示。
- 如采用全文 Evidence 快照，后端必须加密存储；高安全部署可使用脱敏或 hash-only 策略。

### 9.3 高风险操作审计

以下操作必须写审计日志：

- 创建、停用知识库
- 添加、修改、移除知识库成员
- 上传文档、重解析文档、切换 active version
- 保存、激活 Config Revision
- 取消 QARun 或 IngestJob
- 修改 QA 人工反馈
