# AGENTS.md

本文件用于指导 Codex 在当前仓库中的行为。目标是减少常见的 LLM 编码失误：擅自假设、过度设计、无关改动，以及缺少可验证的完成标准。

这些规则默认适用于本仓库全部目录。若更深层目录存在更具体的 `AGENTS.md` 或 `agent.md`，以更具体的文件为准；直接的系统、开发者和用户指令优先级更高。

## 目标

在满足用户需求的前提下：

- 先澄清，再动手
- 优先最简单可行方案
- 只改与任务直接相关的内容
- 用可验证标准判断是否完成

## 1. 编码前先思考

在实现前，先做最小但明确的判断：

- 明确写出关键假设；不确定时，不要偷偷选一种解释然后继续做。
- 如果需求存在多种合理理解，先说明分歧，再选择最稳妥的路径；必要时请求澄清。
- 如果用户要求的方案明显过重、风险更高，或有更简单直接的替代方案，要主动指出。
- 一旦发现信息不足、上下文冲突、接口含义不清、测试预期不明确，先停下并说明问题。

### 执行要求

- 不要把猜测当事实。
- 不要隐藏困惑。
- 不要为了“显得聪明”补做用户没有要求的事。

## 2. 简单优先

始终使用能解决问题的最小实现。

- 不新增用户没有要求的功能。
- 不为了“未来可能会用到”提前抽象。
- 单次使用的逻辑，不要过度封装。
- 不加入没有现实依据的可配置项。
- 不为几乎不可能发生、且当前任务不要求覆盖的场景扩展复杂错误处理。
- 如果 200 行可以压缩到 50 行且可读性更好，应主动简化。

### 自检问题

提交前问自己：

- 这段实现是否比任务本身复杂？
- 一个有经验的工程师会不会认为这是过度设计？
- 是否存在更短、更直接、同样清晰的实现？

如果答案偏向“是”，先简化再继续。

## 3. 外科手术式修改

只改必须改的地方，只清理因本次任务直接产生的问题。

- 不顺手“优化”相邻代码、注释、命名、格式或文件结构。
- 不重构无故障、且与本任务无关的模块。
- 优先遵循现有代码风格、目录结构和约定，即使你个人会用别的写法。
- 如果发现无关的坏味道、历史遗留问题或死代码，可以在结果中说明，但不要擅自删除或重写。

### 允许清理的范围

如果本次改动直接导致以下内容失效，应一并清理：

- 因本次修改而不再使用的 import
- 因本次修改而不再使用的变量、函数、分支
- 因本次修改而失效的测试数据或断言

### 不允许清理的范围

- 任务开始前就已存在的死代码
- 与本次需求无关的格式化调整
- 与本次需求无关的重命名
- 与本次需求无关的依赖升级或目录整理

### 核心判定

每一处改动都应该能直接追溯到用户请求、修复目标，或验证需要。

## 4. 目标驱动执行

不要只照着动词做事，要把任务改写成可验证的结果。

### 任务改写方式

- “加校验” → 先定义哪些非法输入必须失败，再实现并验证
- “修 bug” → 先构造可复现问题的测试或最小复现，再修复并验证
- “重构 X” → 保证重构前后行为一致，并通过现有验证
- “支持新功能” → 明确输入、输出、边界条件和成功标准，再实现

### 多步骤任务

对于非琐碎任务，先给出一个简短计划，并为每一步指定验证方式：

1. [步骤] → 验证: [检查项]
2. [步骤] → 验证: [检查项]
3. [步骤] → 验证: [检查项]

### 完成定义

只有在以下条件满足后，任务才算完成：

- 需求被落实到代码或文档中
- 相关验证已执行，或明确说明无法执行的原因
- 结果与假设、约束、现有风格保持一致
- 没有引入与任务无关的改动

## 5. 修改代码时的具体行为

### 实现前

- 先阅读与任务直接相关的文件，不要只凭文件名猜结构。
- 先确认调用链、数据流、入口点和测试位置。
- 若仓库已有约定，优先沿用现有模式。

### 实现中

- 优先做局部修改，避免扩大影响面。
- 能复用现有函数时，优先复用；但不要为了复用而引入奇怪抽象。
- 除非用户要求，否则不要进行大规模搬移、拆分或架构升级。

### 实现后

- 运行与改动直接相关的检查、测试、构建或 lint。
- 若仓库或上层指令中明确要求某些检查，必须执行。
- 若无法运行某项验证，要明确写出缺失的依赖、环境限制或阻塞原因。

## 6. 沟通方式

与用户沟通时遵循以下原则：

- 先说事实，再说判断。
- 简洁说明假设、权衡、风险和下一步。
- 有歧义时优先提问；若必须在不完整信息下推进，明确说明采用了哪种假设。
- 不夸大结论，不把“可能”表述成“确定”。

## 7. 文档与注释

- 不改动无关注释。
- 不删除看不懂但与任务无关的注释或文档。
- 新增注释只在确有必要时添加，且应解释“为什么”，而不是复述“代码做了什么”。
- 若更新文档，范围仅限与本次改动直接相关的部分。

## 8. 何时主动提醒用户

遇到以下情况，应在结果中明确指出：

- 发现了更简单且风险更低的实现路径
- 用户目标与现有代码结构冲突明显
- 现有测试不足，导致无法高置信验证
- 仓库中存在与本次问题相关、但不在本次修改范围内的潜在隐患

## 9. 轻重缓急

这些规则默认偏向“谨慎优先”，而不是“速度优先”。

但对于非常简单且无歧义的任务，例如：

- 明显的错字修复
- 单行常量修改
- 小范围文案更新

可以减少流程负担，不必机械套用完整步骤；但仍应保持：

- 不乱猜
- 不过度实现
- 不扩大改动范围
- 能验证时就验证

## 10. 产出质量检查清单

提交前至少自查以下项目：

- [ ] 我是否明确了关键假设？
- [ ] 我是否避免了无依据的猜测？
- [ ] 我是否选择了最简单可行方案？
- [ ] 我是否只修改了与任务直接相关的内容？
- [ ] 我是否清理了由本次改动直接产生的无用代码？
- [ ] 我是否运行了相关验证，或说明了无法验证的原因？
- [ ] 我是否能用一句话解释每一类改动为什么存在？

## 11. 判断这些规则是否生效

如果本文件发挥了作用，通常会看到：

- diff 更小、更聚焦
- 返工次数减少
- 澄清问题出现在实现前，而不是出错后
- 代码更短、更直接、更贴近仓库原有风格
- 结果说明里有清晰的验证依据，而不是笼统地说“应该可以”

## 12. 页面设计

页面风格遵循 [DESIGN.md](./DESIGN.md)

## 13. Vibe Coding 协作约定

本项目后续会以 vibe coding 方式推进，但仍必须遵守本文前面的谨慎边界：先澄清、简单优先、外科手术式修改和可验证完成。

### 13.1 开发前阅读顺序

开始编码前，至少先阅读与任务相关的以下文档：

1. [README.md](./README.md)：确认项目结构、本地运行方式和当前已落地依赖。
2. [DESIGN.md](./DESIGN.md)：确认页面设计和视觉约束。
3. [docs/03-系统设计/总体设计说明书.md](./docs/03-系统设计/总体设计说明书.md)：确认系统边界和模块关系。
4. [docs/03-系统设计/详细设计说明书.md](./docs/03-系统设计/详细设计说明书.md)：确认流程、状态和页面到接口映射。
5. [docs/03-系统设计/接口设计说明.md](./docs/03-系统设计/接口设计说明.md)：确认 API、错误码、权限和 DTO 约定。
6. [docs/03-系统设计/数据模型设计.md](./docs/03-系统设计/数据模型设计.md) 与 [docs/03-系统设计/数据库设计.md](./docs/03-系统设计/数据库设计.md)：确认表、索引、状态和数据字典。
7. [docs/04-迭代与交付/编码规范.md](./docs/04-迭代与交付/编码规范.md)：确认命名、代码风格、注释和 Git 规范。

如果任务明显只涉及单个文档或极小范围，可以只阅读直接相关文件，但不能凭文件名猜测现状。

### 13.2 实现前输出最小计划

非琐碎任务开始前，应先用简短中文说明：

- 关键假设。
- 本次会改哪些范围。
- 不会改哪些范围。
- 准备用什么方式验证。

计划应足够短，服务于快速推进；不要把简单任务写成冗长方案。

### 13.3 小步快跑

- 优先完成一条可运行、可验证的最小链路。
- 不新增当前任务没有要求的框架、目录、依赖或配置。
- 不因为“未来可能需要”提前抽象 Provider、Hook、基类或配置项。
- 若需要新增目录或依赖，先说明它为什么是当前任务的最小必要条件。
- 每完成一个backlog，就提一次git commit 并且更新待办文档
- 整个epic做完后 统一进行code review 并且修改
### 13.4 一致性优先

- 文档、代码、接口、数据库设计必须使用同一套业务术语。
- 前端 API 字段使用 `camelCase`，数据库字段使用 `snake_case`。
- `pipelineDefinition` 是后端执行契约，不等同于页面画布 ViewModel。
- PostgreSQL 是业务真值中心，检索副本和图结构不得替代业务真值。
- 发现设计文档之间冲突时，先停下说明冲突点和建议处理方式，不要偷偷选择一种解释继续实现。
